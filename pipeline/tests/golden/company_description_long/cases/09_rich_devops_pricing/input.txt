Shipglass — Feature flags that keep up with your releases

Shipglass is a feature management platform: wrap any change in a flag, roll
it out to 1 percent of users, watch the metrics, and ramp or roll back

Why flags

Shipping behind flags separates deploy from release. Teams merge continuously,
then release deliberately — by cohort, region, or plan. When something breaks,
the kill switch is instant; no revert, no hotfix train.

The alternative is the one every engineer knows: the change is merged, the
deploy is out, and the only way back is another deploy — reviewed,
rebuilt, rolled out — while the bug burns. Flags make release a runtime
decision. The deploy stops being the moment of risk.

How Shipglass works

SDKs for 12 languages evaluate flags locally from a streamed ruleset, so
evaluation adds well under 1 millisecond and works offline. Targeting rules
combine user attributes, cohorts, and percentage rollouts. Every flag change
is audited, and stale flags are surfaced for cleanup automatically. An edge
network keeps rulesets within 50 milliseconds of users worldwide.

The SDKs

Twelve languages, first-party: JavaScript, TypeScript, Python, Go, Java,
Kotlin, Swift, Ruby, PHP, C#, Rust, and Elixir. Every SDK evaluates flags
locally against the streamed ruleset — no network call in the request
path, no flag service to fall over in your critical path. If the stream
drops, the SDK serves the last ruleset it saw and reconciles on
reconnect; a brand-new process falls back to bundled defaults you define.
Server SDKs stream over SSE; mobile SDKs fetch on foreground and cache,
tuned for battery and app-store constraints.

Targeting

Rules combine user attributes, cohorts, and percentage rollouts: 1
percent of free-plan users in Germany, everyone in the beta cohort, 50
percent of new signups. Percentage bucketing is deterministic per user,
so ramps are stable across sessions and services — the user who saw the
feature yesterday sees it today at the same rollout stage. Cohorts sync
from your data warehouse or build from attributes; entitlement flags can
mirror your billing plans so plan gating lives in one place.

Audit and hygiene

Every flag change — who, what, when, from which environment — lands in an
immutable audit log, streamable to your SIEM. Approvals can be required
per environment, so production ramps need a second pair of eyes while
staging stays fast. And because flag debt is the tax every adopter pays,
Shipglass surfaces stale flags automatically: fully ramped for 30 days,
evaluated but never varying, or referenced in code but never evaluated —
each with a cleanup pull request generated for supported languages.

Experiments

Any rollout can become an A/B test: attach a metric, and Shipglass computes
lift with confidence intervals. No separate experimentation stack.

Metrics come from your event stream or warehouse; define a conversion or
numeric metric once and attach it to any flag. The results view shows
lift, confidence intervals, and sample-size progress — enough statistics
to make the call, without a data-science detour. Guardrail metrics watch
error rates and latency during every ramp, whether or not you call it an
experiment: a ramp that degrades a guardrail pauses itself and pages the
flag owner.

Who it's for

Product engineering teams that ship daily. Common first uses: risky
migrations, tiered entitlements, and gradual mobile rollouts where app-store
review makes rollback expensive.

The migration pattern: wrap the new database path in a flag, ramp reads
to 1 percent, compare results in the shadow, ramp to 100, then delete the
old path — with an instant kill switch the whole way. The entitlement
pattern: plan tiers as flags, so packaging changes ship without a deploy.
The mobile pattern: the feature ships dark in the binary, app review
happens once, and the release ramps server-side over days — when a crash
spikes, rollback is a toggle, not a resubmission.

Frequently asked

What happens if Shipglass is down? Your app keeps working. Evaluation is
local; SDKs serve the last streamed ruleset and reconcile when the stream
returns. The edge network exists so rulesets are never far away — within
50 milliseconds of users worldwide.

Does local evaluation leak my targeting rules? Server-side rulesets stay
server-side. Client SDKs receive only evaluated results for the current
user, not the rule set.

Can we self-host? The dedicated environment on Enterprise is
single-tenant; fully self-hosted is on the roadmap and honestly not
close — the edge network is most of the product.

How do you count seats? Humans who log in. Service accounts and CI are
free.

The edge network

Rulesets propagate through Shipglass's edge network — points of
presence on every populated continent, keeping rulesets within 50
milliseconds of users worldwide. A flag change published in the
dashboard reaches every connected SDK in under a second globally,
which is what makes the kill switch a kill switch rather than a
please-stop request. The edge serves rulesets only; evaluation stays
in your process, so the request path never waits on us.

Migration playbook

Coming from a homegrown system: point the SDK at both sources, mirror
your config table into Shipglass flags with the import CLI, and let
the comparison report show divergence before you cut over. Most teams
run dual-read for a week, cut over service by service, and delete the
config table in a month — the audit log and approvals usually retire a
wiki page and a Slack ritual at the same time. Coming from another
flag vendor: the import CLI reads common export formats, and
deterministic bucketing can be seeded to preserve your existing
rollout assignments so no user flips variants mid-migration.

More from the FAQ

Do flags add latency to our requests? No network call happens in the
request path — evaluation is local against the streamed ruleset and
adds well under 1 millisecond.

How do we keep flags from accumulating forever? The stale-flag
detector plus generated cleanup PRs exist for exactly this; teams
that review the stale list in their weekly ritual keep flag counts
flat while shipping more.

Can product managers ramp releases? Yes, within approvals: a PM can
move a ramp through stages the engineering owner defined, with every
change audited and reversible.
