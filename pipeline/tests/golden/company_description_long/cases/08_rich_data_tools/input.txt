Rowfern — Ship data pipelines you can trust

Rowfern is testing and contract enforcement for data pipelines. It catches
broken schemas, silent nulls, and volume anomalies before they reach your
dashboards — in the pull request, not the postmortem.

Star the engine on GitHub. Start a cloud workspace.

What Rowfern does

Rowfern reads your dbt project or SQL pipelines and generates a suite of
checks: schema contracts between producing and consuming models, freshness
and volume expectations learned from history, and assertions you write in
YAML. Checks run in CI on every pull request against a sampled snapshot of
production data, so a column rename or a join that fans out gets caught
before merge. In production, the same checks run on schedule and page the
owning team through PagerDuty or Slack when a contract breaks.

Contracts, not just checks

A test asserts a fact about one table; a contract binds a producer to its
consumers. Rowfern parses your dbt lineage to know which models feed
which, then generates schema contracts at the boundaries: column names,
types, nullability, accepted values. When a pull request changes a
producing model, the contract check runs for every downstream consumer —
the column rename that would have silently nulled a marketing dashboard
becomes a red check on the PR, with the affected consumers listed. The
author sees the blast radius before merge, and consumers get a heads-up
instead of a surprise.

Learned expectations

Freshness and volume checks calibrate themselves from history: how many
rows this table gains on a normal Tuesday, how late its source usually
lands. Anomalies page at the severity you configure, and the baselines
recalibrate as seasonality shifts — a holiday dip is not an incident.
Hand-written YAML assertions cover the business logic history cannot
learn: revenue is non-negative, every order has a customer, the join
keys are unique.

CI integration

Rowfern runs as a check on GitHub and GitLab pull requests. Each run
executes against a sampled snapshot of production data — enough rows to
catch the fan-out and the type error, small enough to finish while the
review is still open. Typical check runs complete in under five minutes.
The PR comment shows exactly which contracts passed, which broke, and who
consumes the broken ones; merge gating is your choice per severity.

Production monitoring

The same checks that gate the PR run on schedule in production — one
definition of correctness, enforced in both places. When a contract
breaks, Rowfern routes the page to the owning team through PagerDuty or
Slack, using the ownership map from the catalog. The alert carries the
failing check, the diff against baseline, and the downstream consumers,
so the on-call engineer starts with the blast radius instead of
discovering it in replies.

Warehouse support

Snowflake, BigQuery, Redshift, and Databricks. Rowfern runs checks inside
your warehouse or over a read-only connection; sampled data never leaves
your environment unless you opt into cloud snapshots.

Docs: Getting started

1. Install the CLI: pip install rowfern.
2. Point it at your dbt project: rowfern init reads your lineage and
   generates a first contract suite in minutes.
3. Add the CI step from the setup page and open a pull request — your
   first contract check runs on the PR.
4. Connect Slack or PagerDuty when you are ready to enforce in
   production.

Most teams start by generating contracts for their most-consumed marts,
tighten from warnings to merge gates over a week or two, then expand
coverage model by model. Nothing requires a big-bang migration.

Frequently asked

How is this different from dbt tests? dbt tests assert facts about
tables; Rowfern binds producers to consumers and enforces the boundary
in CI with production-shaped data, plus learned freshness and volume
baselines in production. The two coexist — Rowfern imports your existing
dbt tests into the catalog.

Will the sampled snapshot leak PII? Sampling runs inside your warehouse;
data leaves only if you opt into cloud snapshots. Column-level masking
rules apply either way.

What does "models under contract" mean for pricing? Models with at least
one enforced contract. Generate contracts everywhere, enforce where it
counts, pay for what is enforced.

Engine versus platform

The open-source engine: contract generation, the full check library,
CI integration, single-project configuration in YAML. The platform
adds what makes it stick across an organization: the browsable contract
catalog, ownership routing for pages, historical baselines computed
from months of runs rather than days, SSO, audit logs, and the in-VPC
read-only deployment. Teams routinely run the engine alone for months
first — that is the funnel working as intended, and the migration to a
workspace imports the existing configuration unchanged.

Docs: Writing assertions

Assertions live in YAML next to the model they protect: not_null with
a tolerance, accepted_values, unique keys, referential integrity
across models, monotonic metrics, and custom SQL predicates for the
business logic only your team knows. Severity levels decide the
consequence — warn annotates the PR, error blocks merge, page routes
through PagerDuty in production. Assertions inherit through lineage,
so a contract on a mart protects the staging models that feed it.

More from the FAQ

Does Rowfern slow down our CI? Check runs execute against sampled
snapshots and typically finish in under five minutes; sampling size is
tunable per project when the trade-off matters.

Who defines ownership? The catalog imports ownership from your dbt
meta tags or a CODEOWNERS-style file; unowned models are flagged,
because an alert without an owner is a notification, not a page.

Can analysts use it, or only engineers? Contracts and assertions are
YAML in the repo, but the catalog is browsable by anyone — consumers
subscribe to the models they depend on and get notified when a
contract on them changes.
